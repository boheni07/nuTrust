# Plan: 관리자 CRUD 관리 기능 (admin-management)

**Feature**: admin-management  
**Phase**: Plan  
**Date**: 2026-04-08  
**Author**: Claude Code (PDCA)

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 고객사·프로젝트 목록 UI가 읽기 전용 골격만 존재해, 관리자가 직접 데이터를 등록·수정·삭제하려면 DB에 직접 접근해야 한다. |
| **Solution** | 모달 기반 CRUD UI를 각 관리 페이지에 추가하고, 고객사 모달에는 부서·담당자 탭을 내장해 하위 자원까지 한 화면에서 관리한다. |
| **Functional UX Effect** | 관리자가 브라우저만으로 고객사·프로젝트·팀·사용자·SLA 정책을 완전히 관리할 수 있어 운영 효율이 크게 향상된다. |
| **Core Value** | API 계층은 이미 완성되어 있으므로 순수 UI 구현으로 즉시 완전한 관리 기능을 확보한다 (리스크 최소). |

---

## Context Anchor

| 항목 | 내용 |
|------|------|
| **WHY** | 서비스 운영 시 고객사·프로젝트 데이터를 UI에서 관리할 수 없어 운영 불가 |
| **WHO** | SYSTEM_ADMIN, MANAGER — 고객사·프로젝트 관리 권한 보유 |
| **RISK** | 삭제 시 연결 티켓 cascade 문제 / 복잡한 탭 모달 UX 일관성 |
| **SUCCESS** | 고객사·프로젝트 CRUD 모달 완동, 하위 자원 탭 동작, 삭제 차단 메시지 |
| **SCOPE** | UI 계층만 — API 계층은 이미 완성 (통합테스트 100% 통과) |

## 2. 요구사항

### FR-01 고객사 등록 모달
- 필드: 고객사명(필수), 이메일(필수), 전화번호(선택), 주소(선택)
- 등록 성공 시 목록 자동 갱신

### FR-02 고객사 수정/상세 모달
- 목록 행 클릭 또는 수정 버튼 → 모달 오픈
- 탭 구성:
  - **기본 정보** 탭: 고객사명, 이메일, 전화, 주소 수정
  - **부서** 탭: 부서 목록 조회, 부서 추가(이름), 부서 삭제
  - **담당자** 탭: 담당자 목록 조회, 담당자 추가(이름, 이메일, 부서, 직책), 삭제

### FR-03 고객사 삭제
- 삭제 버튼 → 확인 다이얼로그
- 연결된 프로젝트/티켓이 있으면 삭제 차단 + 안내 메시지

### FR-04 프로젝트 등록 모달
- 필드: 프로젝트명(필수), 고객사(필수, 드롭다운), 설명(선택), 시작일, 종료일, 상태
- 등록 성공 시 목록 갱신

### FR-05 프로젝트 수정/상세 모달
- 탭 구성:
  - **기본 정보** 탭: 프로젝트명, 고객사, 설명, 날짜, 상태 수정
  - **담당 에이전트** 탭: 배정된 에이전트 목록, 에이전트 추가(드롭다운), 배정 해제

### FR-06 프로젝트 삭제
- 연결된 티켓이 있으면 삭제 차단

### FR-07 팀 수정/삭제
- 팀명 수정 모달
- 팀원 배정: 사용자 목록에서 선택해 팀에 추가
- 팀원 해제: 팀 카드에서 멤버 제거 버튼
- 팀 삭제: 멤버가 없는 팀만 삭제 허용 (또는 확인 후 삭제)

### FR-08 사용자 수정
- 이름, 역할, 팀 변경
- 비밀번호 재설정은 이번 범위 제외

### FR-09 SLA 정책 수정/삭제
- 정책 수정 모달 (기존 생성 폼과 동일 필드)
- 정책 삭제 (기본 정책은 삭제 차단)

## 3. 성공 기준 (Success Criteria)

| SC | 기준 | 측정 방법 |
|----|------|-----------|
| SC-1 | 고객사 등록/수정/삭제 모달이 정상 동작한다 | UI 클릭 테스트, API 응답 201/200/200 |
| SC-2 | 고객사 모달 내 부서/담당자 탭에서 항목 추가/삭제 가능 | 탭 전환 + CRUD 동작 확인 |
| SC-3 | 프로젝트 등록/수정/삭제 모달이 정상 동작한다 | UI 클릭 테스트 |
| SC-4 | 프로젝트 에이전트 배정/해제가 동작한다 | 배정 탭에서 추가/삭제 |
| SC-5 | 티켓 있는 고객사/프로젝트 삭제 시 에러 메시지 노출 | 삭제 시도 → 차단 메시지 확인 |
| SC-6 | 팀 수정/삭제, 팀원 배정/해제 동작 | 팀 관리 페이지 |
| SC-7 | 사용자 수정(이름/역할/팀) 동작 | 사용자 관리 페이지 |
| SC-8 | SLA 정책 수정/삭제 동작 | SLA 관리 페이지 |

## 5. 설계 제약사항

- **기존 코드 재사용**: 기존 `ClientList.tsx`, `ProjectList.tsx` 확장 (새 파일 최소화)
- **모달 컴포넌트**: 재사용 가능한 모달 래퍼 (`ConfirmModal`, `FormModal`) 공통화
- **상태 관리**: React Query `useMutation` 패턴 유지 (기존 코드와 일관성)
- **권한**: SYSTEM_ADMIN, MANAGER만 등록/수정/삭제 가능 (API 계층 이미 처리)
- **삭제 차단**: API에서 409 또는 400 반환 시 UI에서 에러 메시지 표시

## 7. 범위 외 (Out of Scope)

- 고객사 담당자의 유저 계정 연동 생성 (Contact ↔ User 연결)

- 프로젝트별 SLA 정책 연결
- 감사 로그 (audit log)
- 벌크 임포트/익스포트
